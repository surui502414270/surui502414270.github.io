http 抓包实战读书笔记

Fiddler 抓包原理

Fiddler 为什么能抓包？

Fiddler 本质上是一个 Web 代理服务器，它的默认工作端口是 8888

什么是代理服务器？

Web 代理服务器是网络的中间实体，位于 Web 客户端和 Web 服务器之间，扮演“中间人”的角色。http 的代理服务器既是 Web 服务器又是 Web 客户端

对于 Web 客户端来说，代理扮演的是服务器，客户端给代理发请求，代理给客户端发回响应

对于 Web 服务器来说，代理扮演的就是客户端，代理给服务器发请求，服务器给代理发回响应

Fiddler 的工作原理？

在Fiddler启动的时候，会把 Internet 选项中的代理修改为 127.0.0.1，端口：8888。当 Fiddler 退出的时候，它会自动在 Internet 选项中取消代理。如果Fiddler 非正常退出，这是因为 Fiddler 没有自动注销，会造成网页无法访问。需重启 Fiddler

HTTP 协议

常见的 http 方法有哪些？

1.GET 请求指定的页面信息并返回实体主体

2.HEAD 类似于 GET 请求，只不过返回的响应中没有具体的内容，用于获取报头

3.POST 向指定资源提交数据进行处理请求，数据被包含在请求体中，POST 请求可能会导致新的资源的建立或对已有资源的修改

4.PUT 从客户端向服务器传送的数据取代指定文档的内容

5.DELETE 请求服务器删除数据

GET和 POST 的主要区别？

1.GET 提交的数据会放在 URL 之后，以问号分割 URL 和传输数据，参数之间以&相连。POST 方法是把提交的数据放在 http 包中的 Body 里

2.GET 提交的数据大小有限制（因为浏览器对 URL 的长度有限制），而 POST 方法提交的数据大小没有限制

3.GET 方法提交的数据会带来安全问题，比如一个登录页面通过 GET 方式提交数据时，用户名和密码将出现在 URL 上，如果页面可以被缓存或者其他人可以访问这台机器，就可以从历史记录获得该用户的账号密码

http 状态码有哪些？分别是什么作用？

1.200（OK）：请求成功

2.204（No Content 没有内容）：返回的响应中只有一些 Header 和一个状态行，没有实体的主体内容。作用：（1）在不获取资源的情况下了解资源情况（比如判断其类型）（2）通过查看响应中的状态码看某个对象是否存在

3.206（Partial Content 部分内容）：代表服务器已经成功处理了部分 GET 请求（只有发送 GET 方法的 http 请求，服务器才可能返回 206）。作用：（1）像迅雷这种下载工具都是使用 206 状态码来实现断点续传 （2）将一个大文档分解为多个下载段同时下载，一些流媒体技术，比如在线视频可以边看边下载，就是使用 206 状态码来实现的

4.301（Moved Permanently）:请求的页面已经永久性的转移到另一个地址。作用：（1）防止用户输错域名，比如 Google 担心用户输错域名，就把其类似的域名买下来，比如 go0gle，然后重定向到 google.com （2）网站更换域名，比如京东以前的域名是 360buy.com,现在的域名是 jd.com

5.302（Found）：临时重定向。比如：在未登录的情况下，访问需要登录才能访问的页面，就会被服务器返回 302

6.304（Not Modified）：上次的文档已经被缓存了，还可以继续使用

7.400（Bad Request）：客户端请求有语法错误，如参数不正确

8.401（Unauthorized）：未授权，有些网页采用的是 http 基本认证，需要在 http 请求中带上 Authorization Header，否则服务器会返回 401

9.403（Forbidden）：请求被拒绝

10.404：找不到资源

11.500（Internal Server Error）：服务器内部错误，比如程序内部抛出异常、空指针等

12.503（Server Unavailable）：服务器暂时不可用，由于服务器维护或过载，服务器当前无法处理请求

http 的 header 包含哪些关键字？

1.Cookies：它由key=value 的形式组成，浏览器把 Cookie 通过 http 请求中的“Cookie：header”发送给 Web 服务器，Web 服务器通过 http 响应中的“Set-Cookie：header”把 Cookie 发送给浏览器

2.Accept：表示客户端浏览器可以接受的媒体类型，例如 Accept：text/html 代表浏览器可以接受服务器返回 html

3.Accept-Encoding：跟压缩有关，如 Accept-Encoding：gzip，deflate（告诉服务器，浏览器支持 zip 压缩）

4.Accept-Language：浏览器声明自己接受的语言

5.User-Agent:浏览器告诉服务器，客户端使用的操作系统及版本、CPU 类型、浏览器及版本、浏览器渲染引擎、浏览器语言、浏览器插件等

6.Referer：让服务器判断来源页面，即用户是从哪个页面来的

7.Connection：从 http/1.1起，系统都默认开启了 Connection：Keep-Alive，保持连接特性。http 协议是基于 tcp 协议的，当一页网页完全打开后，客户端和服务器之间用于传输 http 数据的 tcp 连接不会关闭，如果客户端再次访问这个服务器上的页面，将会继续使用这一条已经建立的连接。Keep-Alive 不会永久保持连接，它有一个保持时间，可以在不同的服务器软件中设定这个时间

8.Host:指定被请求的主机和端口号

如何判断缓存新鲜度？

Web 服务器通过以下 2 种方式来判断浏览器缓存是否最新

1.通过缓存文件的最后修改时间来判断缓存的新鲜度。如果客户端想请求一个文档，它首先检查本地缓存，发现存在这个文档的缓存，获取缓存中文档的最后修改时间，通过“If-Modified-Since”发送 http 请求给服务器。服务器收到请求，将服务器的文档修改时间（Last-Modified）跟If-Modified-Since作比较，如果一样，说明缓存还是最新的，服务器将发送状态码 304 客户端。如果不一样，则服务器发送最新的文档给客户端

2.ETag，是根据实体内容生成的一段 hash 字符串（类似于 MD5或者 SHA1 之后的结果），可以标志资源的状态，当资源发生改变的时候，ETag 也随之变化。使用 ETag 主要是为了解决一些 Last-Modified 无法解决的问题，如某些服务器不能精确得到文件的最后修改时间、文件的修改非常频繁，Last-Modified只能精确到秒、一些文件的最后修改时间改变但内容并未改变

Cookie 分为哪两类？

1.会话 Cookie：一种临时的 Cookie,它记录了用户访问站点时的设置和偏好，关闭浏览器，会话 cookie 就被删除了

2.持久 Cookie ：存储在硬盘上，不管浏览器退出或者计算机重启，持久 cookie 都继续存在，持久 cookie 有过期时间

软件测试读书笔记

1.2 软件缺陷是什么

至少满足下列5个规则之一才称发生了一个软件缺陷

  1. 软件未实现产品规格说明书要求的功能
  2. 软件出现了产品规格说明书指明不应该出现的错误
  3. 软件实现了产品规格说明书未提到的功能
  4. 软件未实现产品规格说明书虽未明确提及但应该实现的目标
  5. 软件难以理解、不易使用、运行缓慢、或者（从测试员的角度看）最终用户会认为不好

1.5 软件测试员究竟做些什么

软件测试员的目标：尽可能早地找出缺陷，并确保其得以修复

2.3 软件开发生命周期模式

螺旋模式：一开始不必详细定义所有的细节，先定义重要功能，努力实现这些功能，接受客户反馈，然后进入下一阶段

  1. 确定目标、可选方案和限制条件
  2. 明确并化解风险
  3. 评估可选方案
  4. 当前阶段开发和测试
  5. 计划下一阶段
  6. 确定进入下一阶段的方法

软件测试员喜欢这个模式，因为他们很早就参与开发过程，有机会尽早发现问题，为项目节省时间和金钱

4.1 黑盒、白盒、静态、动态测试

  1. 黑盒测试（功能测试、行为测试）：只需知道软件要做什么，无法知道软件是如何运行的
  2. 白盒测试：可以访问程序的代码，并通过检查代码来协助测试
  3. 静态测试：只是检查和审核
  4. 动态测试：使用和运行软件

5.1 动态黑盒测试：带上眼罩测试软件

  1. 输入数据->接受输出->检验结果
  2. 动态黑盒测试又称行为测试，因为测试的是软件在使用过程中的实际行为
  3. 测试用例：进行测试时使用的特定输入，以及测试软件的过程步骤
  4. 等价类划分是指分步骤地把海量测试用例缩减得很小，但过程同样有效
  5. 数据测试等价类划分原则：边界条件；次边界条件；默认、空白、空值和无；非法、错误、不正确和垃圾数据
  6. 状态转换图应该表示出：软件可能进入的每一种独立状态；从一种状态进入另一种状态的输入和条件。进入或者退出状态的设置条件及输出结果
  7. 通过性状态测试：检查软件、描绘状态、尝试各种合法可能性、确认状态及其转换正常
  8. 失效性状态测试：竞争条件和时序错乱、重复、压迫和重负

6.1 检查代码

  1. 静态白盒测试是在不执行软件的条件下有条理地仔细审查软件设计、体系结构和代码，从而找出软件缺陷的过程，有时也称为结构化分析
  2. 正式审查是进行静态白盒测试的过程，包括4个基本要素：确定问题；遵守规则；准备；编写报告
  3. 编码标准和规范：可靠性；可读性/维护性；移植性
  4. 通用代码审查清单：数据引用错误（导致缓冲区溢出，安全缺陷）；数据声明错误；计算错误；比较错误；控制流程错误；子程序参数错误；输入/输出错误；其他检查

7.1 动态白盒测试：带上X光眼镜测试软件

动态白盒测试：查看代码功能（做什么）和实现方式（怎么做）得到的信息来确定哪些需要测试、哪些不需要测试、如何展开测试。（软件测试员可以查看并使用代码的内部结构，从而设计和执行测试，了解软件的运作方式有助于选择合适的测试方法。）

  1. 单元测试/模块测试、集成测试、系统测试
  2. 递增测试两条途径：自底向上（测试驱动调用被测试模块）和自顶向下（测试桩）
  3. 数据覆盖：数据流、次边界、公式和等式、错误强制
  4. 代码覆盖：程序语句和代码行覆盖、分支覆盖、条件覆盖

8 配置测试

  1. 确定所需硬件类型
  2. 确定有哪些厂商的硬件、型号和驱动程序可用
  3. 确定可能的硬件特性、模式和选项
  4. 将确定后的硬件配置缩减为可控制的范围
  5. 明确与硬件配置有关的软件唯一特性
  6. 设计在每一种配置中执行的测试用例
  7. 在每种配置中执行测试
  8. 反复测试直到小组对结果满意为止

9 兼容性测试

软件兼容性测试是指检查软件之间是否能够正确地交互和共享信息
兼容性测试记住3点：

  1. 对兼容软件的所有可能选择进行等价划分
  2. 研究适用于测试软件的高级/低级标准和规范
  3. 测试软件程序之间的不同数据流动方式

11 易用性测试

优秀UI具备的7个要素：符合标准和规范；直观；一致；灵活；舒适；正确；实用

13 软件安全性测试

黑客动机：挑战/成名，好奇，借用，恶意破坏（丑化、破坏、拒绝服务），偷窃

14 网站测试

  1. 黑盒测试：文本（当作文档对待）；超级链接；图片；表单（用于输入和选择信息的文本框、列表框和其他域）；对象和其他各种简单功能
  2. 灰盒测试（介于白盒和黑盒之间）适合网页测试，HTML（超文本标记语言）
  3. 白盒测试，网站系统结构和编程知识：动态内容；数据库驱动的网页；用编程方法创建的网页；服务器性能和加载；安全性
  4. 配置和兼容性测试，易用性测试

15 自动测试和测试工具

  1. 重复执行测试的过程称为回归测试
  2. 工具和自动化的主要属性：速度；效率；准确度和精确度；节省资源；仿真和模拟；坚持不懈
  3. 软件测试自动化可以执行测试用例，查找软件缺陷，分析看到的信息，记录结果
  4. 自动化：宏录制和回放；可编程的宏；完全可编程的自动测试工具
  5. 随机测试：模拟用户可能的操作，测试猴子

pgsql常用命令

1.启动 PostgreSQL：

pg_ctl -D /usr/local/var/postgres -l /usr/local/var/postgres/server.log start

2.关闭 PostgreSQL：

pg_ctl -D /usr/local/var/postgres stop -s -m fast

3.创建一个 PostgreSQL 用户

createuser username -P

4.创建数据库

createdb dbname -O username -E UTF8 -e

5.连接数据库

psql -U username -d dbname -h 127.0.0.1

6.显示已创建的数据库：

\l

7.连接数据库

\c dbname

8.显示数据库表

\d

9.创建一个名为 test 的表

CREATE TABLE test(id int, text VARCHAR(50));

10.插入一条记录

INSERT INTO test(id, text) VALUES(1, ‘sdfsfsfsdfsdfdf’);

11.查询记录

SELECT * FROM test WHERE id = 1;

12.更新记录

UPDATE test SET text = ‘aaaaaaaaaaaaa’ WHERE id = 1;

13.删除指定的记录

DELETE FROM test WHERE id = 1;

14.删除表

DROP TABLE tablename;

15.删除数据库

DROP DATABASE dbname;

前端总结

目录结构

Vue

  • Vue生命周期

共分为八个阶段
beforeCreate（实例刚被创建）el 和 data 并未初始化
created（实例创建完成，属性已绑定，但DOM 还未生成，$el属性还不存在）
beforeMount(模板编译/载入前)完成了 el 和 data 初始化 。此时还没有生成 html到页面上去
mounted（模板编译/载入后）完成挂载，也就是模板中的 html 渲染到 html 页面中，此时一般可以做一些 ajax 操作
beforeUpdate（数据更新时调用）
updated（更新完成后的钩子）
beforeDestroy（实例销毁之前调用。在这一步，实例仍然完全可用）
destroyed（Vue 实例销毁后调用）

  • Vue路由

    抽取共用的部分，比如导航栏，做主路由。之后再分子路由，如在论坛页面，将导航栏和用户信息抽取出来做公共部分放在 bbs/index 下。子路由分为 home、newart、hotart等，实现点击切换子路由。
    动态路由：如 article/：id，帖子具体页面，根据url 的 id 不同，得到不同数据。

  • Vuex

    Vuex 是一个专为 Vue.js 应用程序开发的状态管理模式。它采用集中式存储管理应用的所有组件的状态，并以相应的规则保证状态以一种可预测的方式发生变化。

    Getter：Vuex 允许我们在 store 中定义“getter”（可以认为是 store 的计算属性）。就像计算属性一样，getter 的返回值会根据它的依赖被缓存起来，且只有当它的依赖值发生了改变才会被重新计算。
    Mutation：更改 Vuex 的 store 中的状态的唯一方法是提交 mutation。Vuex 中的 mutation 非常类似于事件：每个 mutation 都有一个字符串的 事件类型 (type) 和 一个 回调函数 (handler)。
    Action类似于 mutation，不同在于：Action 提交的是 mutation，而不是直接变更状态。Action 可以包含任意异步操作。

  • 基于 Token的身份验证–JWT
    JWT就是一个字符串，经过加密处理与校验处理的字符串，形式为：A.B.C
    A由JWT头部信息header加密得到
    B由JWT用到的身份验证信息json数据加密得到
    C由A和B加密得到，是校验部分
    A.B.C就是生成的 token 了
    token应用流程？
    初次登录：用户初次登录，输入用户名密码
    密码验证：服务器从数据库取出用户名和密码进行验证
    生成JWT：服务器端验证通过，根据从数据库返回的信息，以及预设规则，生成JWT
    返还JWT：服务器的HTTP RESPONSE中将JWT返还
    带JWT的请求：以后客户端发起请求，HTTP REQUEST HEADER中的Authorizatio字段都要有值，为JWT

  • 路由拦截
    第一套 :
    第一层 meta是否控制路由需要验证
    第二层 store中token是否存在
    第三层 localstore中token是否存在，存在则进行token登录，成功将获得新token 存store
    第二套 :
    第一层 meta是否控制路由需要验证
    第二层 localstore中token是否存在

2.中间件

当我们谈论Iris中的中间件时，我们正在讨论在HTTP请求生命周期中的主要处理程序代码之前和/或之后运行代码。例如，日志记录中间件可能会将传入的请求细节写入日志，然后调用处理程序代码，然后编写有关日志响应的详细信息。关于中间件的一件很酷的事情是这些单元非常灵活且可重用。中间件只是一个Handler形式func(ctx iris.Context)，中间件在以前的中间件调用时正在执行ctx.Next()，这可以用于身份验证，即：如果登录，则会ctx.Next()引发错误响应。
